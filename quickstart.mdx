<Steps>

<Step title="Add the library to your project">

The library is **header-only** — there is nothing to compile and no runtime to link. Add it to your CMake project using either `FetchContent` or `add_subdirectory`.

<Tabs>
  <Tab title="FetchContent">

```cmake
include(FetchContent)

FetchContent_Declare(
    baremetal-async
    GIT_REPOSITORY https://github.com/intel/cpp-baremetal-senders-and-receivers.git
    GIT_TAG        main  # or a specific commit/tag
)
FetchContent_MakeAvailable(baremetal-async)

target_link_libraries(my_target PRIVATE async)
```

  </Tab>
  <Tab title="add_subdirectory">

Clone the repository alongside your project, then add it as a subdirectory:

```bash
git clone https://github.com/intel/cpp-baremetal-senders-and-receivers.git extern/async
```

```cmake
add_subdirectory(extern/async)
target_link_libraries(my_target PRIVATE async)
```

  </Tab>
</Tabs>

<Note>
  CMake 3.25 or later is required. The library sets `cxx_std_20` as a minimum compile feature via the `async` target; use the `main` branch and set `CMAKE_CXX_STANDARD 23` for C++23 support.
</Note>

</Step>

<Step title="Include the headers you need">

Each facility lives in its own header under `<async/...>`. Include only what you use:

```cpp
#include <async/just.hpp>        // async::just — produces values immediately
#include <async/then.hpp>        // async::then — transforms values
#include <async/sync_wait.hpp>   // async::sync_wait — blocks until completion
```

There is no single umbrella header; this keeps compile times low on resource-constrained toolchains.

</Step>

<Step title="Write your first sender chain">

A sender chain describes a unit of asynchronous work. Building the chain does **no** work — work only starts when you call a consumer such as `async::sync_wait`.

```cpp
#include <async/just.hpp>
#include <async/then.hpp>
#include <async/sync_wait.hpp>

int main() {
    // Describe the computation: produce 42, then double it.
    auto sndr = async::just(42)
              | async::then([](int i) { return i * 2; });

    // Start the chain and wait for the result.
    // Returns std::optional<stdx::tuple<int>>
    auto result = async::sync_wait(sndr);

    auto [value] = result.value();
    // value == 84
}
```

<Tip>
  The `|` pipe operator is the idiomatic way to compose senders. Each adaptor on the right-hand side wraps the sender on the left-hand side. No work happens until a consumer starts the chain.
</Tip>

</Step>

<Step title="Fire and forget with start_detached">

For operations that run to completion without needing a result back in the calling scope, use `async::start_detached`. This is the typical pattern in interrupt-driven firmware where you launch async work in response to a hardware event and do not block waiting for it.

```cpp
#include <async/just.hpp>
#include <async/then.hpp>
#include <async/start_detached.hpp>

void on_hardware_event(int raw) {
    auto sndr = async::just(raw)
              | async::then([](int v) { process(v); });

    // Starts the chain; returns immediately.
    async::start_detached(sndr);
}
```

<Warning>
  The operation state must remain alive for the duration of the async work. `async::start_detached` handles this through the configured allocator policy. See the allocator documentation for details on controlling storage for detached operations.
</Warning>

</Step>

<Step title="Build and run">

Build and run as you would any CMake project:

```bash
cmake -B build -DCMAKE_CXX_STANDARD=23
cmake --build build
./build/my_target
```

For a cross-compiled bare metal target, supply your toolchain file as usual:

```bash
cmake -B build \
    -DCMAKE_TOOLCHAIN_FILE=path/to/arm-none-eabi.cmake \
    -DCMAKE_CXX_STANDARD=23
cmake --build build
```

</Step>
